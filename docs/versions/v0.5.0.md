# Typetchi v0.5.0 規格：跨分頁狀態同步與設定區簡化

## 1. 版本目標

v0.5.0 是 Typetchi 的跨分頁狀態同步與設定區簡化版本。

目前已完成：

* Chrome Extension 基礎架構
* Content Script 注入
* Shadow DOM 隔離
* 不干擾宿主頁面
* chrome.storage.local 儲存
* Extension context invalidated 防護
* 大型 SPA 相容性
* Widget 拖拉、縮放、固定、收合
* 角色動畫與互動回饋
* EXP、Level、三階段進化
* 今日輸入統計
* CPM / WPM 統計
* 貼上文字不計入 EXP
* 重置角色進度
* Widget 版面穩定修正

本版本主要解決：

1. A 分頁輸入或操作 Typetchi 後，B 分頁狀態不會即時同步。
2. B 分頁需要重新整理才會看到最新狀態。
3. 固定位置、收合、關閉、尺寸、角色 EXP、Level、Stage 等狀態需要跨所有分頁同步。
4. 「重置視窗位置」按鈕不需要保留。
5. 「重置角色進度」不需要做得太明顯，應移到較低調的設定區或危險操作區。

---

## 2. 本版本核心任務

v0.5.0 需要完成：

1. 建立跨分頁 state sync 機制。
2. 任一分頁更新 petState，其他分頁即時同步。
3. 任一分頁更新 widgetState，其他分頁即時同步。
4. 任一分頁更新 settings，其他分頁即時同步。
5. 固定位置、收合、關閉、尺寸等 Widget 狀態需同步。
6. 角色 EXP、Level、Stage、今日輸入、今日最高速度需同步。
7. 不需要重新整理頁面即可看到最新狀態。
8. 移除「重置視窗位置」按鈕。
9. 降低「重置角色進度」在 UI 上的顯眼程度。
10. 避免跨分頁同步造成無限 storage 寫入循環。
11. 保持不干擾宿主網站。

---

## 3. 本版本不做項目

v0.5.0 不做以下功能：

```txt
不做分支進化
不做多角色選擇
不做任務系統
不做成就系統
不做排行榜
不做雲端同步
不做 AI 對話
不做後端 API
不做完整打字練習模式
不做跨裝置同步
```

本版本只做同一個 Chrome profile 內的分頁同步，不做不同電腦或不同 Chrome 使用者之間的同步。

## 4. 同步策略總覽

### 4.1 優先使用 chrome.storage.onChanged

跨分頁同步的主機制：

```txt
chrome.storage.local
chrome.storage.onChanged
```

同步流程：

```txt
A 分頁狀態更新
→ A 分頁寫入 chrome.storage.local
→ Chrome 觸發 chrome.storage.onChanged
→ B / C / D 分頁收到 storage change
→ 各分頁更新自己的 React state
→ UI 即時同步
```

### 4.2 tabs 事件作為補強

可選擇在 background service worker 監聽：

```txt
chrome.tabs.onActivated
chrome.tabs.onUpdated
```

用途：

```txt
切換分頁時，可通知當前 active tab 重新讀取 storage
新頁面完成載入時，可讓 content script 初始化最新狀態
```

但 v0.5.0 的主要同步不應依賴 tabs event。
tabs event 只作為補強，因為不是只有切換分頁時才需要同步，任一分頁開著時都應該能透過 storage change 更新。

```ts
  WIDGET_STATE: 'typetchi.widgetState',
  SETTINGS: 'typetchi.settings',
  TYPING_STATS: 'typetchi.typingStats',
} as const;
```

若目前已有 key，請以現有架構為準，但要集中管理。

## 7. Storage Change Listener

### 7.1 建立 Hook

請新增：

```txt
src/hooks/useStorageSync.ts
```

用途：

```txt
監聽 chrome.storage.onChanged
當指定 key 變更時，同步更新 React state
```

### 7.2 基本範例

```ts
import { useEffect } from 'react';
import { STORAGE_KEYS } from '../storage/storageKeys';

export function useStorageSync(params: {
  onPetStateChanged?: (nextState: UserPetState) => void;
  onWidgetStateChanged?: (nextState: WidgetState) => void;
  onSettingsChanged?: (nextState: SettingsState) => void;
  onTypingStatsChanged?: (nextState: TypingStatsPersistedState) => void;
}) {
  useEffect(() => {
    if (!globalThis.chrome?.storage?.onChanged) {
      return;
    }

    const handleChanged = (
      changes: Record<string, chrome.storage.StorageChange>,
      areaName: string
    ) => {
      if (areaName !== 'local') return;

      const petChange = changes[STORAGE_KEYS.PET_STATE];
      if (petChange?.newValue && params.onPetStateChanged) {
        params.onPetStateChanged(petChange.newValue);
      }

      const widgetChange = changes[STORAGE_KEYS.WIDGET_STATE];
      if (widgetChange?.newValue && params.onWidgetStateChanged) {
        params.onWidgetStateChanged(petChange.newValue);
      }

      const settingsChange = changes[STORAGE_KEYS.SETTINGS];
      if (settingsChange?.newValue && params.onSettingsChanged) {
        params.onSettingsChanged(settingsChange.newValue);
      }

      const typingStatsChange = changes[STORAGE_KEYS.TYPING_STATS];
      if (typingStatsChange?.newValue && params.onTypingStatsChanged) {
        params.onTypingStatsChanged(typingStatsChange.newValue);
      }
    };

    chrome.storage.onChanged.addListener(handleChanged);

    return () => {
      chrome.storage.onChanged.removeListener(handleChanged);
    };
  }, [
    params.onPetStateChanged,
    params.onWidgetStateChanged,
    params.onSettingsChanged,
    params.onTypingStatsChanged,
  ]);
}
```

注意：上方範例中請確認 widgetChange 實際呼叫的是 `widgetChange.newValue`，不得誤用 `petChange.newValue`。

## 8. 避免同步回寫循環

### 8.1 問題

如果 A 分頁寫入 storage，B 分頁收到變更後 setState，然後 B 分頁又自動 save storage，可能造成：

```txt
A 寫入
B 收到
B setState
B save
A 收到
A setState
A save
...
```

需避免無限循環。

---

### 8.2 解法：區分 local update 與 remote update

請在 state update flow 中標記來源：

```ts
export type StateUpdateSource = 'local' | 'remote';
```

當使用者在本分頁操作：

```txt
source = local
更新 React state
debounce save storage
```

當 storage.onChanged 觸發：

```txt
source = remote
只更新 React state
不得再次觸發 save storage
```

---

### 8.3 實作方向

可在 hooks 中使用 ref：

```ts
const isApplyingRemoteUpdateRef = useRef(false);

function applyRemoteWidgetState(nextState: WidgetState) {
  isApplyingRemoteUpdateRef.current = true;
  setWidgetState(nextState);
  queueMicrotask(() => {
    isApplyingRemoteUpdateRef.current = false;
  });
}

function handleLocalWidgetChange(nextState: WidgetState) {
  setWidgetState(nextState);

  if (!isApplyingRemoteUpdateRef.current) {
    scheduleSaveWidgetState(nextState);
  }
}
```

或在 reducer action 中帶 source：

```ts
dispatch({
  type: 'widget/set',
  payload: nextState,
  source: 'remote',
});
```

remote action 不應觸發 storage save。

## 9. 狀態版本與更新時間

### 9.1 建議增加 updatedAt

為了處理分頁競態，建議每個持久化 state 加上：

```ts
updatedAt: number;
```

例如：

```ts
export interface PersistedWidgetState extends WidgetState {
  updatedAt: number;
}
```

每次 local save 時更新：

```ts
updatedAt: Date.now()
```

收到 remote state 時，若 remote.updatedAt 比本地舊，則忽略。

```ts
if (nextState.updatedAt < currentState.updatedAt) {
  return;
}
```

---

### 9.2 不要為 runtime-only state 加 updatedAt

以下不需要持久化或加 updatedAt：

```txt
recentCpm
recentWpm
sessionChars
typing events
animationState
speechBubble visible
expToast visible
```

## 10. 切換分頁事件補強

### 10.1 背景

使用者提到：

```txt
我記得可以抓取到切換分頁的事件
```

確實可以在 background service worker 透過 tabs event 做補強。

### 10.2 建議用途

當 active tab 切換時，background 可傳 message 給該 tab，要求 content script 重新讀取 storage。

流程：

```txt
chrome.tabs.onActivated
→ chrome.tabs.sendMessage(tabId, { type: 'TYPETCHI_SYNC_FROM_STORAGE' })
→ content script 收到
→ reload persisted state from chrome.storage.local
→ update React state
```

### 10.3 注意

此功能需要 `tabs` permission 或 active tab 條件，請檢查 manifest 權限。

若目前不想增加 `tabs` permission，可以先不做此補強，只靠 `chrome.storage.onChanged`。

v0.5.0 建議：

```txt
優先完成 storage.onChanged
tabs.onActivated 作為 optional enhancement
避免為了這版增加不必要權限
```

## 11. 初始載入同步

每個分頁注入時，必須從 storage 讀取最新狀態。

流程：

```txt
content script injected
→ load petState from storage
→ load widgetState from storage
→ load settings from storage
→ load typingStats persisted state from storage
→ normalize
→ set React state
→ start storage.onChanged listener
```

如果 storage unavailable，沿用 v0.1.2 的 fallback 行為。

### 12.5 關閉同步

操作：

```txt
A 分頁點擊關閉
```

預期：

```txt
B 分頁同步關閉
C 分頁同步關閉
```

注意：

若目前 `closed` 不持久化，v0.5.0 需要重新確認產品規則。

本版本需求是：

```txt
關閉狀態需要同步到所有分頁。
```

因此可以新增：

```ts
settings.widgetVisible = boolean
```

或：

```ts
widgetState.closed = boolean
```

建議使用：

```ts
settings.widgetVisible
```

理由：

```txt
closed 比較像暫時 UI 狀態
widgetVisible 比較像使用者設定
```

最低要求：

```txt
A 分頁關閉後，其他分頁也要關閉
重新整理頁面後是否維持關閉，依目前產品需求決定
```

建議 v0.5.0 行為：

```txt
關閉 = hide widget globally until user opens from extension popup or reload setting
收合 = collapse widget globally but保留 handle
```

如果目前沒有 popup 入口，請不要做「永久完全關閉到無法恢復」。

因此 v0.5.0 建議先採用：

```txt
關閉只同步到目前已開啟分頁，不永久保存
收合才持久化保存
```

若必須永久保存關閉狀態，請提供可恢復入口，例如 extension popup 中的「顯示 Typetchi」。

## 17. 多分頁同時輸入的行為

### 17.1 可能情境

A 分頁與 B 分頁幾乎同時輸入文字。

### 17.2 最低要求

v0.5.0 不要求完美處理所有競態，但需避免明顯覆蓋錯誤。

### 17.3 建議策略

Pet EXP 更新請採用「讀取最新 storage 後合併」或「增量事件」方式。

#### 較簡單方案

每次有效打字時：

```txt
1. 從目前 local state 計算 next petState
2. 寫入 storage
3. 其他分頁透過 onChanged 同步
```

此方案簡單，但若兩個分頁完全同時輸入，可能發生後寫覆蓋前寫。

#### 較穩方案

將打字 EXP 更新封裝為增量 apply：

```ts
async function addTypingExp(deltaExp: number) {
  const latestPetState = await loadPetState();
  const nextPetState = applyTypingExp(latestPetState, deltaExp);
  await savePetState(nextPetState);
  setPetState(nextPetState);
}
```

v0.5.0 建議至少在打字 EXP 寫入前讀一次 latest storage，降低覆蓋機率。

## 20. scripts/content-dist.js 注意事項

本專案目前 build 流程會將：

```txt

```

複製成：

```txt
dist/content/index.js
```

因此如果 v0.5.0 有修改 content script runtime 行為，例如：

```txt

storage.onChanged
tabs message handling
runtime message handling
widget state sync
pet state sync
settings sync
```

請務必同步更新：

```txt
src/

```

並在執行：

```txt

```

```txt
dist/content/index.js
```

包含最新修正。

若本次只改 React component 且 build 流程會正常 bundle React component，可不直接手動修改 `scripts/content-dist.js`。

但若目前專案的 content script 實際仍依賴 `scripts/content-dist.js` 手寫輸出，請務必同步。

## 21. 測試案例

### 21.1 Pet state 跨分頁同步

1. 開啟 A 分頁與 B 分頁。
2. 兩個分頁都顯示 Typetchi。
3. 在 A 分頁手打 10 字。
4. A 分頁 EXP 增加。
5. B 分頁不重新整理，也應同步 EXP。
6. B 分頁 Level / Stage 若有變化，也應同步。

---

### 21.2 Widget 位置同步

1. 開啟 A 分頁與 B 分頁。
2. 在 A 分頁拖曳 Widget 到新位置。
3. 停止拖曳後，B 分頁 Widget 位置同步更新。
4. 不需要重新整理 B 分頁。

---

### 21.3 Widget 尺寸同步

1. 開啟 A 分頁與 B 分頁。
2. 在 A 分頁縮放 Widget。
3. 停止縮放或 debounce 後，B 分頁 Widget 尺寸同步更新。
4. 不需要重新整理 B 分頁。

---

### 21.4 固定狀態同步

1. A 分頁點擊固定。
2. B 分頁同步固定。
3. A 分頁解除固定。
4. B 分頁同步解除固定。

---

### 21.5 收合狀態同步

1. A 分頁點擊收合。
2. B 分頁同步收合。
3. B 分頁點擊展開。
4. A 分頁同步展開。

---

### 21.6 關閉狀態同步

1. A 分頁點擊關閉。
2. B 分頁同步關閉或隱藏。
3. 若沒有 popup 恢復入口，不得讓使用者永久無法重新顯示 Typetchi。
