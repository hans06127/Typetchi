# Typetchi v0.6.4 Bugfix：Collapsed Handle 拖曳底部邊界修正

## 1. 版本目標

v0.6.4 是 Typetchi 的收合狀態拖曳 bugfix 版本。

目前 v0.6.3 已修正：

- 收合後不再殘留大型模糊 div
- 收合後畫面只剩小 icon / handle
- close 行為仍維持完全消失
- popup 可重新顯示 Typetchi

但目前仍有新問題：

- 收合後的小 icon / handle 無法拖曳到瀏覽器最底部
- 拖到底部後會自動彈回中間偏下的位置
- 看起來像是拖曳邊界或 normalize 邏輯仍使用完整 Widget 高度，而不是 handle 高度

本版本只處理 collapsed handle 的拖曳邊界與位置保存問題。

---

## 2. 本版本處理範圍

v0.6.4 只修正：

1. 收合後小 icon / handle 的拖曳邊界。
2. 收合後小 icon / handle 可拖曳到 viewport 底部。
3. 收合後小 icon / handle 放開後不得彈回中間。
4. collapsed handle 的 normalize 必須使用 handle 尺寸。
5. expanded widget 的 normalize 與 collapsed handle 的 normalize 要分開處理。
6. 重新整理頁面後，collapsed handle 的底部位置應保留。
7. 跨分頁同步後，其他分頁不得把 handle 位置修回錯誤高度。

v0.6.4 不做以下功能：

- 不修改 evolution system
- 不修改 EXP / Level / Stage
- 不修改貼上文字不計入 EXP 規則
- 不修改 CPM / WPM
- 不修改每日統計

- 不新增任務系統
- 不新增成就系統
- 不新增角色美術
- 不做 UI 大改版

除非上述功能直接影響 collapsed handle 的位置計算，否則不要調整。

## 4. 問題描述

目前收合後的小 icon 可以拖曳，但拖曳到底部附近後，放開會自動回到畫面中間偏下的位置。

預期行為應該是：

- 使用者拖到哪裡，小 icon 就停在哪裡
- 只要位置仍在 viewport 可視範圍內，就不應被修回中間
- 小 icon 應能接近 viewport 四邊
- 底部邊界應根據 handle 高度計算，而不是完整 Widget 高度

---

## 5. 推測原因

請檢查以下可能原因：

1. collapsed 狀態仍使用完整 Widget height 計算拖曳邊界。
2. y 座標 clamp 使用了 expanded widget height。
3. collapsed handle 與 expanded widget 共用同一個 normalize function，但沒有區分尺寸。
4. 拖曳時使用 top-left x/y，保存或載入時卻被轉成 bottom/right。
5. storage onChanged 收到遠端 state 後，又套用 expanded widget normalize。
6. 重新整理時 loadWidgetState normalize 把 collapsed handle 位置當成 expanded widget 位置處理。
7. root host / shadow container 的座標系與 viewport 座標混用。
8. handle 實際尺寸與程式使用的 handle 尺寸不一致。

---

## 6. 開發方向

### 6.1 拖曳邊界依 Widget 狀態分開

請依狀態分開計算拖曳範圍：

- normal / expanded：使用完整 Widget width / height
- collapsed：使用 collapsed handle width / height
- closed：不需要拖曳計算

collapsed 狀態不得用完整 Widget 高度限制 handle 的 y 座標。

---

### 6.2 collapsed handle 可以拖到 viewport 底部

collapsed handle 的底部最大位置應接近：

- viewport height
- 扣掉 handle height
- 扣掉安全 margin

不得使用：

- viewport height 扣掉完整 Widget height

因為這會導致 handle 只能拖到畫面中間偏下。

---

### 6.3 collapsed normalize 與 expanded normalize 分離

請將位置 normalize 分成兩種語意：

1. collapsed handle normalize  
   用於小 icon / handle，尺寸應以 handle 為準。

2. expanded widget normalize  
   用於完整 Widget，尺寸應以 Widget width / height 為準。

不要讓 collapsed handle 被 expanded widget 的尺寸規則修正。

---

### 6.4 儲存位置格式需一致

請確認目前 widget position 的儲存格式。

需要檢查：

- 是 x / y
- 還是 right / bottom
- 是否使用 transform translate
- 是否有不同狀態使用不同定位方式

如果拖曳時計算的是 x / y，保存與載入也應保持一致。

不要出現：

- 拖曳用 x / y
- 保存用 bottom / right
- 載入時又當成 x / y

這會造成放開或重新載入後位置錯誤。

---

### 6.5 拖曳後展開的位置處理

收合狀態下，小 icon 可以拖到 viewport 底部。

當使用者點擊 handle 展開時：

- 完整 Widget 需要出現在合理可視位置
- 如果完整 Widget 在底部放不下，可以在展開時再 normalize expanded widget
- 但不應反過來限制 collapsed handle 不能放到底部

也就是：

- collapsed 時按照 handle 尺寸保存位置
- expanded 時再按照完整 Widget 尺寸做顯示修正

## 8. 測試方向

### 8.1 底部拖曳測試

測試流程：

1. 開啟 Typetchi。
2. 點擊收合。
3. 拖曳小 icon / handle 到瀏覽器最底部附近。
4. 放開滑鼠。
5. 小 icon / handle 應停留在底部附近。
6. 不應彈回中間偏下的位置。

---

### 8.2 四邊拖曳測試

測試流程：

1. 收合 Typetchi。
2. 拖曳 handle 到左上角。
3. 拖曳 handle 到右上角。
4. 拖曳 handle 到左下角。
5. 拖曳 handle 到右下角。

預期：

- 每個位置都應能停留在可視範圍內。
- handle 不應超出 viewport。
- handle 不應被錯誤拉回中間。

---

### 8.3 重新整理後位置測試

測試流程：

1. 收合 Typetchi。
2. 將 handle 拖到底部附近。
3. 重新整理頁面。
4. 檢查 handle 位置。

預期：

- handle 仍應停留在底部附近。
- 不應回到中間偏下。
- 不應被 expanded widget normalize 修正。

---

### 8.4 跨分頁同步測試

測試流程：

1. 開啟 A 分頁與 B 分頁。
2. A 分頁收合 Typetchi。
3. A 分頁將 handle 拖到底部附近。
4. 觀察 B 分頁。

預期：

- B 分頁的 handle 位置也同步到底部附近。
- B 分頁不應自動修回中間偏下。
- A / B 分頁位置應一致或合理接近。

---

### 8.5 點擊與拖曳分離測試

測試流程：

1. 單純點擊 handle。
2. Widget 應展開。
3. 再次收合。
4. 拖曳 handle 超過拖曳門檻。
5. 放開滑鼠。

預期：

- 單純點擊會展開。
- 拖曳不應誤觸展開。
- 拖曳只更新位置。

---

### 8.6 展開後位置測試

測試流程：

1. 收合 Typetchi。
2. 將 handle 拖到底部附近。
3. 點擊 handle 展開。

預期：

- 完整 Widget 需顯示在合理可視位置。
- 如果完整 Widget 因高度無法完全放在底部，允許展開時向上修正。
- 但收合狀態的 handle 位置不應因此永久被修回中間。

---

### 8.7 Close 回歸測試

測試流程：

1. 點擊關閉。
2. Typetchi 應完全消失。
3. 不得留下 handle。
4. 點 Chrome extension popup 的「顯示 Typetchi」。
5. Typetchi 應重新顯示。

---
