## v0.1.2 補充修正：Extension Context Invalidated 與 Widget 顯示修正

### 1. 問題背景

開發期間曾出現以下錯誤：

```txt
Uncaught (in promise) Error: Extension context invalidated.
```

同時也出現過：

```txt
[Typetchi] content script loaded
[Typetchi] root injected
[Typetchi] shadow root attached
[Typetchi] React app mounted
[Typetchi] root ensured
[Typetchi] storage loaded
[Typetchi] typing tracked
```

但畫面仍未顯示小視窗。

這代表 Typetchi 的 content script、root injection、React mount 與 typing tracker 都有執行，問題不是「沒有載入」，而是：

1. 舊 content script 在 extension reload 後仍嘗試呼叫失效的 Chrome Extension API。
2. widget 狀態或 shadow host 樣式導致 UI 被藏起來或裁切。
3. 實際執行的 content script 是由 `scripts/content-dist.js` 複製到 `dist/content/index.js`，因此只修改 `src/` 不足以修正實際 extension 行為。

### 3. Storage 防護修正

#### 修改檔案

```txt
src/storage/extensionStorage.ts
scripts/content-dist.js
```

#### 修正內容

新增 storage disabled 狀態：

```ts
let isStorageDisabled = false;
```

新增 guarded Chrome storage accessor：

```ts
function getChromeLocalStorage() {
  try {
    if (isStorageDisabled) return null;
    return globalThis.chrome?.storage?.local ?? null;
  } catch (error) {
    disableStorage(error);
    return null;
  }
}
```

當 storage 讀寫失敗，或讀取 `globalThis.chrome.storage.local` 本身就丟出 `Extension context invalidated` 時，呼叫：

```ts
disableStorage(error);
```

後續所有 storage get / set 都直接 fallback 或 no-op，不再碰 `chrome.storage.local`。

---

### 4. Storage get / set 行為

#### getStorageValue

失敗時：

```txt
不 throw error
不產生 unhandled promise rejection
回傳 defaultValue
設定 isStorageDisabled = true
```

#### setStorageValue

失敗時：

```txt
不 throw error
不產生 unhandled promise rejection
直接 no-op
設定 isStorageDisabled = true
```

#### content-dist callback 版本

`storageGet()` 行為：

```txt
storage 不可用 → resolve(fallback)
chrome.runtime.lastError → disableStorage(error) → resolve(fallback)
exception → disableStorage(error) → resolve(fallback)
```

`storageSet()` 行為：

```txt
storage 不可用 → resolve()
chrome.runtime.lastError → disableStorage(error) → resolve()
exception → disableStorage(error) → resolve()
```

### 5. 實際執行檔同步修正

本專案的 build 流程會將：

```txt
scripts/content-dist.js
```

複製成：

```txt
dist/content/index.js
```

因此 content script runtime 相關修正不可只修改 `src/`。

凡是涉及以下功能，都需要確認 `scripts/content-dist.js` 是否同步更新：

```txt
storage get / set
typing tracker
root injection
shadow root
widget visibility
runtime guard
extension context invalidated handling
```

驗收時需確認：

```txt
npm run build
```

後產出的：

```txt
dist/content/index.js
```

包含本次修正。

### 6. Widget State 正規化

#### 修改檔案

```txt
src/storage/widgetStorage.ts
```

#### 修正內容

新增：

```ts
normalizeWidgetState()
```

目的：

```txt
避免舊 storage 中的 widget x / y / width / height 超出 viewport
避免舊 storage 中的 closed: true 導致 reload 後完全不顯示 widget
```

載入 widget state 時：

```txt
x / y / width / height 需 clamp 回 viewport 內
closed 強制 reset 為 false
```

保存 widget state 時也需經過 normalize，避免非法狀態再次寫回 storage。

---

### 7. 關閉按鈕行為修正

#### 修改檔案

```txt
src/components/PetWidget/PetWidget.tsx
```

#### 修正內容

關閉按鈕不再永久保存：

```txt
closed: true
```

避免使用者或舊版程式把 `closed: true` 寫進 storage 後，未來 reload 頁面時 widget 直接不顯示。

新的行為：

```txt
關閉只影響目前 React state
不透過 storage flush path 永久保存 closed: true
```

後續若要做正式「隱藏 Typetchi」功能，應另外設計成明確設定，例如：

```ts
settings.widgetEnabled = false;
```

不要與暫時關閉 / 收合狀態混用。

---

### 8. Shadow Host 顯示修正

#### 修改檔案

```txt
src/content/injectRoot.ts
```

#### 修正內容

調整 root host isolation style。

原因：

```txt
shadow host 原本若使用固定尺寸或 paint containment
fixed-position widget 在 shadow DOM 內可能被 host 裁切
導致 console 顯示 mounted 但畫面不可見
```

修正方向：

```txt
host 使用 inset: 0
host width / height 設為 0
host overflow: visible
移除可能造成裁切的 paint containment
```

目標：

```txt
Typetchi root 不參與宿主頁面 layout
Typetchi widget 不被 shadow host 裁切
Typetchi UI 仍維持 fixed positioning
透明區域不影響宿主頁面點擊
```

---

### 9. Extension Reload 後的行為

Chrome Extension 在開發期間 reload 後，舊頁面上的 content script 可能仍留在舊 JS execution context，但 extension runtime / API context 已失效。

本版本修正後：

```txt
舊頁面不應持續噴 unhandled Extension context invalidated
storage 失敗後會 disabled
後續狀態可暫時 fallback 到 in-memory
```

但若要恢復完整 storage 功能，仍需要重新整理該網頁。

預期行為：

```txt
不重新整理舊頁面：
Typetchi 盡量維持目前頁面記憶體狀態，但不保證 storage 可用。

重新整理舊頁面：
content script 重新注入，取得新的 extension context，storage get / set 恢復正常。
```

### 11. 驗收條件

本修正完成後需符合：

1. 開發期間 reload extension 後，舊頁面不應持續噴出 unhandled `Extension context invalidated`。
2. storage get 失敗時回傳 fallback，不 throw。
3. storage set 失敗時 no-op，不 throw。
4. 第一次 storage 失敗後，後續不再繼續碰失效的 `chrome.storage.local`。
5. `src/storage/extensionStorage.ts` 與 `scripts/content-dist.js` 都有同步修正。
6. `npm run build` 後，`dist/content/index.js` 包含最新 storage guard。
7. 舊 storage 若含有 `closed: true`，reload 後 widget 仍應顯示。
8. widget x / y / width / height 若超出 viewport，需被修正回可視範圍。
9. shadow host 不得裁切 fixed-position widget。
10. Console 顯示 React mounted 時，畫面也應實際可見。
