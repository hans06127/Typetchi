# v0.6.0 補充規格：關閉恢復與版本號同步修正

## 1. 補充目標

v0.6.0 除了進化系統重構，也需要先修正兩個目前已發現的產品問題：

1. 點擊關閉後，Typetchi 無法再開啟。
2. Chrome 擴充套件載入頁面仍顯示版本 `0.1.2`，疑似版本號沒有隨版本更新。

這兩個問題優先於進化系統重構，因為它們會直接影響測試、驗收與使用者操作。

---

# 2. 關閉功能修正

## 2.1 目前問題

目前點擊關閉按鈕後，Typetchi 會消失，但使用者沒有明確方式可以再開啟。

可能原因：

```txt
closed = true 被保存到 storage
widgetState.closed 跨分頁同步後所有分頁都關閉
React render 時直接 return null
沒有保留 handle / bubble / popup 入口
沒有從 extension icon 重新顯示的功能
```

結果是：

```txt
按下關閉
→ Widget 消失
→ 沒有任何入口可以恢復
→ 使用者只能重整、清 storage 或重新載入 extension
```

這在產品行為上不可接受。

---

## 2.2 v0.6.0 關閉行為重新定義

v0.6.0 請暫時不要做「永久完全關閉」。

在沒有 extension popup 或明確恢復入口之前，關閉按鈕不得讓 Typetchi 完全消失到無法恢復。

請將目前的關閉行為改成以下其中一種，優先採用方案 A。

---

## 2.3 方案 A：關閉等同收合到 handle

### 行為

點擊右上角關閉按鈕時，不要讓 React `return null`，也不要把 widget 完全移除。

請改成：

```txt
關閉按鈕
→ 設定 collapsed = true
→ 保留小型 handle / bubble
→ 使用者點擊 handle 可以重新展開
```

也就是：

```txt
close button = collapse to handle
```

使用者看到的行為：

```txt
點擊關閉
→ 小視窗向下收合
→ 留下一個小頭像 / handle
→ 點擊 handle 可重新開啟
```

### 優點

```txt
不需要 popup
不會讓使用者卡死
符合目前已有的收合邏輯
跨分頁同步也比較安全
```

---

## 2.4 方案 B：真正關閉，但需要恢復入口

如果仍要保留真正的 close / hide 功能，則必須同時新增恢復入口。

最低要求：

```txt
點擊 Chrome extension icon popup
→ 顯示「顯示 Typetchi」按鈕
→ 點擊後把 widgetVisible 改回 true
→ 所有分頁同步重新顯示 Typetchi
```

如果 v0.6.0 不打算新增 popup，請不要採用方案 B。

---

# 3. closed state 處理規則

## 3.1 不得讓 closed 造成永久不可恢復

v0.6.0 請檢查所有與下列欄位有關的邏輯：

```txt
closed
widgetState.closed
settings.widgetVisible
```

若目前有：

```tsx
if (widgetState.closed) {
  return null;
}
```

請改掉。

不得在沒有恢復入口的情況下直接 `return null`。

---

## 3.2 建議規則

v0.6.0 建議採用：

```txt
collapsed：可持久化、可跨分頁同步
closed：不持久化，或 migration 時強制 false
widgetVisible：暫不使用，除非有 popup 恢復入口
```

### 建議

```txt
關閉按鈕 → setCollapsed(true)
收合按鈕 → setCollapsed(true)
handle 點擊 → setCollapsed(false)
```

也就是先把 close / collapse 整合為同一種可恢復行為。

---

## 3.3 舊資料 migration

舊 storage 可能已經存入：

```json
{
  "closed": true
}
```

v0.6.0 載入 widgetState 時必須修正：

```txt
closed 一律 normalize 成 false
如果 collapsed 不存在，預設 false
```

範例：

```ts
export function normalizeWidgetState(rawState: unknown): WidgetState {
  const normalized = {
    ...DEFAULT_WIDGET_STATE,
    ...safeParsedState,
  };

  return {
    ...normalized,
    closed: false,
  };
}
```

目的：

```txt
避免使用者因舊 storage 的 closed: true 導致 Typetchi 永久不顯示
```

---

## 3.4 UI 文案調整

如果按鈕實際行為是收合，請不要叫「關閉」。

建議改成：

```txt
收合
```

或 icon tooltip：

```txt
收合小視窗
```

若未來有真正關閉功能，才使用：

```txt
隱藏 Typetchi
```

---

## 3.5 跨分頁同步規則

v0.6.0 中：

```txt
collapsed 需要跨分頁同步
closed 不建議跨分頁持久化
```

行為：

```txt
A 分頁點擊收合 / 關閉按鈕
→ A 分頁 collapsed = true
→ B 分頁同步 collapsed = true
→ 所有分頁都保留 handle
→ 任一分頁點擊 handle 展開
→ 所有分頁 collapsed = false
```

---

## 3.6 關閉功能驗收條件

完成後需符合：

1. 點擊右上角關閉 / 收合按鈕後，Typetchi 不會永久消失。
2. 點擊後至少會保留 handle / bubble / 小頭像。
3. 點擊 handle 可以重新展開。
4. 重新整理頁面後不會因舊的 `closed: true` 導致 Widget 消失。
5. 跨分頁同步後，其他分頁也不會進入無法恢復狀態。
6. 程式中不得在沒有恢復入口時因 `closed === true` 直接 `return null`。
7. Tooltip 或文案需符合實際行為，若是收合就不要叫真正關閉。

---

# 4. 擴充套件版本號同步修正

## 4.1 目前問題

Chrome 擴充套件載入頁面仍顯示版本：

```txt
0.1.2
```

但目前專案已進行到：

```txt
v0.6.0
```

這代表版本號可能沒有同步更新。

可能原因：

```txt
manifest.json version 沒改
dist/manifest.json 是舊的
build script 複製了舊 manifest
package.json version 有更新但 manifest 沒同步
docs 版本有更新但 extension manifest 沒同步
Chrome 載入的是舊 dist 目錄
Chrome extension 頁面沒有按重新載入
```

---

## 4.2 v0.6.0 版本號要求

v0.6.0 完成時，以下位置版本號需一致：

```txt
package.json
manifest.json
dist/manifest.json
docs/v0.6.0.md
```

目標版本：

```txt
0.6.0
```

注意：

```txt
Chrome extension manifest 的 version 應使用 0.6.0
Git tag / PR title 可以使用 v0.6.0
```

---

## 4.3 檢查 manifest 來源

請檢查專案實際 build 流程。

可能情況：

### 情況 A：根目錄 manifest.json 直接複製到 dist

請確認：

```txt
manifest.json version = 0.6.0
npm run build 後 dist/manifest.json version = 0.6.0
```

### 情況 B：manifest 由 scripts 產生

請檢查：

```txt
scripts/build-dist.mjs
scripts/manifest-template.json
scripts/content-dist.js
```

若 build script 寫死版本號，請移除硬寫，改從 package.json 讀取版本，或至少同步更新成 0.6.0。

### 情況 C：Chrome 載入錯目錄

請確認 Chrome 的「載入未封裝項目」指向的是最新 build 目錄。

常見錯誤：

```txt
你 build 到 dist/
但 Chrome 載入的是專案根目錄或舊的 dist copy
```

---

## 4.4 建議新增版本同步檢查

請新增或調整 build 流程，讓 manifest version 不容易忘記更新。

建議方式：

```txt
以 package.json version 作為唯一來源
build 時產生 dist/manifest.json
dist/manifest.json.version = package.json.version
```

如果目前不想改 build script，至少要加入檢查。

---

## 4.5 建議新增 npm script

可以新增：

```json
{
  "scripts": {
    "check:version": "node scripts/check-version.mjs"
  }
}
```

`scripts/check-version.mjs` 檢查：

```txt
package.json version
manifest.json version
dist/manifest.json version
```

若不一致，直接 exit 1。

---

## 4.6 簡易 check-version 行為

檢查項目：

```txt
1. 讀取 package.json version
2. 讀取 manifest.json version
3. 若 dist/manifest.json 存在，也讀取 dist/manifest.json version
4. 三者不一致時印出錯誤
5. 一致時印出：
   Typetchi version check passed: 0.6.0
```

---

## 4.7 build 後驗收方式

完成 v0.6.0 後請執行：

```txt
npm run typecheck
npm run build
```

如果有新增 script：

```txt
npm run check:version
```

然後確認：

```txt
dist/manifest.json
```

裡面的：

```json
{
  "version": "0.6.0"
}
```

---

## 4.8 Chrome 測試步驟

請依照以下方式確認版本號：

```txt
1. 執行 npm run build
2. 開啟 chrome://extensions
3. 找到 Typetchi
4. 按下重新載入 extension
5. 若仍顯示舊版本，移除 extension
6. 重新載入 dist/ 目錄
7. 確認卡片顯示 0.6.0
```

如果仍顯示 0.1.2，代表 Chrome 載入的不是最新 dist，或 dist/manifest.json 仍是舊版。

---

## 4.9 版本號驗收條件

完成後需符合：

1. `package.json` version 為 `0.6.0`。
2. `manifest.json` version 為 `0.6.0`。
3. `dist/manifest.json` version 為 `0.6.0`。
4. `npm run build` 後不會產出舊版 manifest。
5. Chrome extension 管理頁顯示版本 `0.6.0`。
6. 若新增 `npm run check:version`，版本不一致時會失敗。
7. 文件、PR title、manifest version 不再互相脫節。

---

# 5. v0.6.0 任務優先順序調整

v0.6.0 請依以下順序執行：

```txt
1. 修正版本號同步，確認 Chrome 顯示 0.6.0。
2. 修正關閉後無法恢復問題。
3. 修正舊 storage 中 closed: true 的 migration。
4. 確認收合 / 展開 / 跨分頁同步正常。
5. 再進行 evolution system 重構。
6. 執行完整回歸測試。
```

---

# 6. 補充驗收條件

v0.6.0 除原本進化系統驗收條件外，還需新增：

1. 點擊關閉 / 收合後，Typetchi 仍可透過 handle 重新展開。
2. 不會因 `closed: true` 造成永久消失。
3. 舊 storage 含有 `closed: true` 時，載入後會 normalize 成可顯示狀態。
4. 跨分頁收合 / 展開同步正常。
5. Chrome extension 管理頁顯示版本 `0.6.0`。
6. `package.json`、`manifest.json`、`dist/manifest.json` 版本一致。
7. `npm run build` 後不會回到舊版 `0.1.2`。
8. 若有 `check:version` script，需通過。
